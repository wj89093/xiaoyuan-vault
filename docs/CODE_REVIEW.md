# 晓园 Vault 代码审查标准与流程

> 版本：v1.0
> 更新：2026-04-30
> 适用范围：晓园 Vault 全部源代码（main / preload / renderer）

---

## 一、为什么需要代码审查

| 维度 | 价值 |
|------|------|
| **质量** | 及早发现 bug、安全漏洞和性能问题 |
| **一致性** | 确保代码风格和架构决策统一 |
| **知识共享** | 审查过程即团队学习过程，避免知识孤岛 |
| **可维护性** | 保证代码可读、可测试、可扩展 |

---

## 二、审查流程

### 2.1 流程总览

```
开发完成 → 自查 → 提交 PR → 自动化检查 → 人工审查 → 修改 → 合并
           │                      │              │
           └── 自查清单 ──┘     └── CI 门禁 ──┘  └── 审查清单
```

### 2.2 详细步骤

| 步骤 | 角色 | 说明 | 时限 |
|------|------|------|------|
| 1. 开发完成 | 作者 | 功能开发完成，本地测试通过 | — |
| 2. 自查 | 作者 | 按自查清单逐项检查（见第四章） | 提交前 |
| 3. 提交 PR | 作者 | 填写 PR 模板，关联 Issue，指定审查人 | — |
| 4. 自动化检查 | CI | ESLint / TypeScript / 测试 / 构建 | ≤ 10 分钟 |
| 5. 人工审查 | 审查人 | 按 Checklist 审查，逐项标注 | ≤ 2 个工作日 |
| 6. 修改 | 作者 | 按审查意见修改，重新提交 | ≤ 1 个工作日 |
| 7. 合并 | 审查人 | 所有审查项通过，Approve 后合并 | — |

### 2.3 分级审查

根据变更范围和风险等级，审查要求不同：

| 等级 | 触发条件 | 审查人 | 要求 |
|------|----------|--------|------|
| **P0 — 严格** | 主进程核心逻辑、数据库、安全相关、IPC 接口变更 | ≥ 2 人 | 全量 Checklist + 手动测试 |
| **P1 — 标准** | 新功能、组件变更、API 对接 | ≥ 1 人 | 全量 Checklist |
| **P2 — 简化** | 文档、样式调整、配置变更 | ≥ 1 人 | 精简 Checklist（类型安全 + 规范） |

## 三、审查维度与权重

每个审查维度配有权重，审查时按权重分配精力：

| 维度 | 权重 | 关注点 |
|------|------|--------|
| **正确性** | 30% | 逻辑是否正确、边界条件、错误处理 |
| **类型安全** | 20% | 无 any、类型完整、泛型正确 |
| **安全** | 15% | IPC 校验、路径遍历、输入净化、敏感信息 |
| **可维护性** | 15% | 命名、结构、复杂度、注释 |
| **性能** | 10% | 渲染优化、内存泄漏、不必要的重渲染 |
| **测试** | 10% | 测试覆盖、边界用例、测试质量 |

## 四、代码审查 Checklist

### 4.1 自查清单（作者提交 PR 前必检）

> 作者在提交 PR 前必须逐项确认，未通过的项不得提交。

#### 类型安全

- [ ] 无 `any` 类型（必须使用 `unknown` + 类型守卫）
- [ ] 所有函数有显式返回类型注解
- [ ] 可选属性用 `?` 标注，访问时使用可选链 `?.`
- [ ] `interface` 优先于 `type`（除非需要联合类型）
- [ ] 导入类型使用 `import type`

#### 规范合规

- [ ] 命名符合规范（camelCase / PascalCase / UPPER_SNAKE）
- [ ] 文件名符合规范（组件 PascalCase.tsx / 非组件 kebab-case.ts）
- [ ] 导入顺序：内置 → 第三方 → 相对 → 类型
- [ ] 使用函数组件 + Hooks（无类组件）
- [ ] 常量使用 `const` + UPPER_SNAKE 命名

#### 错误处理

- [ ] async 函数有 try-catch
- [ ] catch 块中有 `log.error()` 且包含上下文信息
- [ ] 自定义错误类型继承 `AppError`
- [ ] 不吞异常（空 catch 块需注释说明原因）

#### 安全

- [ ] IPC handler 有参数类型校验
- [ ] 文件路径有路径遍历检查（禁止 `..`）
- [ ] 不使用 `eval()` / `new Function()`
- [ ] 日志中无敏感信息（API Key、密码等）

#### 测试

- [ ] 新功能有对应测试
- [ ] 边界条件和异常路径有覆盖
- [ ] 本地 `npm test` 通过
- [ ] 本地 `npm run lint` 无错误

---

### 4.2 审查清单（审查人使用）

> 审查人按此清单逐项审查，每项标注 ✅ 通过 / ⚠️ 建议改进 / ❌ 必须修改。

#### A. 正确性（30%）

| # | 检查项 | 说明 |
|---|--------|------|
| A1 | 逻辑正确 | 核心逻辑是否正确实现需求 |
| A2 | 边界条件 | 空数组、null、undefined、0、空字符串等 |
| A3 | 错误路径 | 异常是否有合理的 fallback / 用户提示 |
| A4 | 数据流 | 状态变更是否有完整的闭环（加载 → 成功/失败） |
| A5 | 竞态条件 | 异步操作是否有并发问题（快速连续调用） |

#### B. 类型安全（20%）

| # | 检查项 | 说明 |
|---|--------|------|
| B1 | 无 any | 搜索 `any` 关键字，全部替换为具体类型 |
| B2 | 返回类型 | 所有导出函数有显式返回类型 |
| B3 | 泛型正确 | 泛型约束合理，无过度泛化 |
| B4 | 类型断言 | `as` 断言有安全保障（前有类型检查） |
| B5 | 枚举/字面量 | 状态值使用字面量联合类型或 enum |

#### C. 安全（15%）

| # | 检查项 | 说明 |
|---|--------|------|
| C1 | IPC 校验 | 主进程 IPC handler 有参数类型校验 |
| C2 | 路径安全 | 文件操作有路径遍历防护 |
| C3 | 输入净化 | 用户输入有 XSS/注入防护 |
| C4 | 敏感信息 | 无硬编码密钥、日志脱敏 |
| C5 | 权限最小化 | renderer 通过 preload 受控访问，不直接暴露 Node API |

#### D. 可维护性（15%）

| # | 检查项 | 说明 |
|---|--------|------|
| D1 | 命名清晰 | 变量/函数名自解释，无歧义缩写 |
| D2 | 函数长度 | 单函数 ≤ 50 行，超出则拆分 |
| D3 | 嵌套深度 | ≤ 3 层，超出用提前返回/卫语句 |
| D4 | 注释质量 | 解释"为什么"而非"是什么" |
| D5 | 职责单一 | 每个模块/函数只做一件事 |
| D6 | 重复代码 | 无明显 copy-paste，提取公共函数 |

#### E. 性能（10%）

| # | 检查项 | 说明 |
|---|--------|------|
| E1 | 重渲染 | React 组件无不必要的重渲染（memo / useCallback） |
| E2 | 内存泄漏 | useEffect 清理订阅、定时器、事件监听 |
| E3 | 大列表 | 长列表使用虚拟滚动 |
| E4 | 懒加载 | 大组件/库使用 React.lazy / dynamic import |
| E5 | 数据库查询 | SQL 查询有索引支持，无 N+1 问题 |

#### F. 测试（10%）

| # | 检查项 | 说明 |
|---|--------|------|
| F1 | 覆盖率 | 新代码测试覆盖率 ≥ 80% |
| F2 | 边界用例 | 测试覆盖 null / 空值 / 异常路径 |
| F3 | 测试隔离 | 测试间无依赖，可独立运行 |
| F4 | Mock 合理 | Mock 最小化，不 Mock 被测逻辑本身 |

## 五、审查意见规范

### 5.1 意见分类

审查意见必须标注类型，避免歧义：

| 标签 | 含义 | 处理方式 |
|------|------|----------|
| `❌ MUST` | 必须修改，不修改不合并 | 作者必须修改或给出合理反驳 |
| `⚠️ SHOULD` | 建议修改，但不阻塞合并 | 作者可选择修改或说明理由 |
| `💡 IDEA` | 改进建议，仅供参考 | 作者自行决定是否采纳 |
| `❓ QUESTION` | 疑问，需要解释 | 作者回复说明 |

### 5.2 意见格式

```
[标签] [检查项编号] 简要描述

详细说明（可选）：
- 为什么有问题
- 建议的修改方式
- 相关代码示例
```

**示例**：

```
❌ MUST B1: 发现 any 类型

src/main/services/database.ts:189 使用了 `any`：
  normalizeRecord(r: any)

建议替换为具体类型或使用 unknown + 类型守卫：
  normalizeRecord(r: Record<string, unknown>)
```

### 5.3 审查意见原则

1. **对事不对人** — 评论代码，不评论人
2. **说清楚为什么** — 不只说"这里不好"，要解释为什么不好
3. **给出方案** — 最好附带建议的修改方式
4. **区分轻重** — MUST 和 SHOULD 不要混用
5. **及时审查** — 收到 PR 后 2 个工作日内完成审查

## 六、常见问题速查

### 6.1 类型安全类

| 问题 | 正确做法 | 错误示例 |
|------|----------|----------|
| 使用 `any` | `unknown` + 类型守卫 | `function fn(x: any)` |
| 缺少返回类型 | 显式标注 `: Promise<void>` | `async function fn()` |
| 类型断言无保障 | 先检查再断言 | `data as SomeType` |
| IPC 参数无校验 | 运行时 `typeof` 检查 | 直接使用 `ipcMain.handle` 参数 |

### 6.2 安全类

| 风险 | 检查方式 | 修复方案 |
|------|----------|----------|
| 路径遍历 | 搜索 `path.join` / `readFile` | 添加 `..` 检查 |
| XSS | 搜索 `dangerouslySetInnerHTML` | 使用 React 渲染或 DOMPurify |
| 硬编码密钥 | 搜索 `sk-` / `api_key` / `password` | 环境变量 + `.env` |
| 敏感日志 | 搜索 `log.info` / `log.error` | 脱敏处理 |

### 6.3 性能类

| 问题 | 检测方式 | 修复方案 |
|------|----------|----------|
| 不必要重渲染 | React DevTools Profiler | `React.memo` / `useCallback` |
| 内存泄漏 | 搜索 `addEventListener` / `setInterval` | `useEffect` cleanup |
| 大包体积 | `npm run build` 检查产物大小 | 懒加载 / tree-shaking |

## 九、特殊场景处理

### 9.1 紧急修复（Hotfix）

- 可跳过人工审查，但必须事后补审
- 合并后 24 小时内完成补审
- 补审发现的问题必须立即修复

### 9.2 大规模重构

- 先提交 RFC（设计文档）审查
- 拆分为多个小 PR，每个 PR ≤ 400 行变更
- 每个PR 独立可测试、可回滚

### 9.3 意见分歧

1. 审查人与作者先在 PR 中讨论
2. 无法达成一致时，由项目负责人裁决
3. 裁决结果记录在 PR 评论中
