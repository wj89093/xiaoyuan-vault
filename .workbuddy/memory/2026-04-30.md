- 测试文件需要单独配置 parserOptions（继承 type-checked 的 config）

## 文件导入功能 Bug 修复

### Bug 1: 文件移到 raw files/ 而非 0-收集/
- `file:import` handler 将文件 rename 到 `vaultPath/raw files/`，用户期望直接进 `0-收集/`
- 修复：将目标目录从 `raw files/` 改为 `0-收集/`

### Bug 2: convertWithJS 使用已被移走的原路径
- 文件 rename 到 dest 后，`canConvertWithJS(filePath)` 和 `convertWithJS(filePath)` 仍用原路径
- 原路径已不存在（ENOENT），导致所有可转换格式的转换都会失败
- 修复：去掉转换逻辑，导入就是纯移动文件到 `0-收集/`

### Bug 3: 拖拽文件 File.path 在 contextIsolation 下为 undefined
- Electron 34 + `contextIsolation: true` 下，渲染进程的 `File` 对象没有 `.path` 属性
- `e.dataTransfer.files` 中的 File 是标准 Web File，缺少 Node.js 扩展的 path
- 修复：在 preload 中暴露 `webUtils.getPathForFile(file)`，ImportApp 用它获取文件路径

### 用户要求
- 导入 = 移动文件到 `0-收集/`（原位置文件消失）
- `0-收集/` 中的文件永远保留，不删
- 不需要格式转换

### 修改的文件
- `src/main/index.ts` — `file:import` handler：改为 rename 到 `0-收集/`，去掉转换逻辑
- `src/main/index.ts` — `import:saveUrl` handler：`raw files/` → `0-收集/`
- `src/preload/index.ts` — 新增 `getPathForFile` 暴露 `webUtils.getPathForFile`
- `src/renderer/ImportApp.tsx` — 用 `getPathForFile` 替代 `file.path`，增加错误提示
